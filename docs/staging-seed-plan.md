# LeagueXI Post-WC — Staging Seed Plan (QA data)

> Deterministic test data for the UI QA checklist, on a **migrated STAGING**
> Supabase project + `post-wc` Vercel preview. **Never production / never live.**
> This is a plan — nothing is executed until approved.

## Form factor
A **Node/TS seed script** (`scripts/seed-staging.ts`, run with `tsx`) using the
**staging service-role key**. SQL alone can't create `auth.users` safely, so user
creation goes through the Supabase **Admin API**; everything else is table inserts +
two RPC calls. The script is **not** wired into the app and never imported by it.

## Why a script + RPCs (faithful, not hand-faked)
After inserting fixtures/predictions/scores it calls the **real** pipeline:
- `recalculate_match_predictions(F_done)` → scores predictions (5/3/0).
- `recalculate_leaderboards(R_open)` → writes Round + Season + per-league
  Round/Season `leaderboard_entries`.
So leaderboards/scoring are exercised authentically; All-Time is query-time over the
season rows (no separate insert). **No cron needed.**

## 1. Seed data model & exact tables affected
Baseline already present from migrations (do **not** re-seed / never delete on reset):
`prediction_contexts` (active standard 2026-27), `seasons` (2026-27), `teams` (15
clubs), Global League, RLS/RPCs.

The seed writes/affects:
| Table | What the seed adds |
|---|---|
| `auth.users` (Admin API) | 4 users (qa, comp1, comp2, admin) |
| `profiles` | usernames set (auto-creates via trigger); `is_admin=true` for admin |
| `seasons` | + **2027-28** (status `upcoming`) — needed for the context-create rejection test |
| `competitions` | 1 staging competition (NOT NULL FK target for fixtures) |
| `leaguexi_rounds` | **R_open** (status `open`) + **R_next** (status `draft`, future) |
| `fixtures` | 4 in R_open (all four states) + 2 in R_next; `is_included=true`; `logo_url` stays null |
| `predictions` | QA/comp1/comp2 picks (see §3) |
| `leagues` | **L_private** (private) + **L_public** (public), creator = QA |
| `league_members` | QA+comp1+comp2 in both (Global auto-joined via username trigger) |
| `leaderboard_entries` | generated by `recalculate_leaderboards` (Round/Season/League) |
| `sync_logs` | a few rows incl. an old success + failures (stale/consecutive UI) |
| `system_alerts` | 1 **unresolved** + 1 **resolved** alert (resolve UI + unread badge) |

All seeded rows use **fixed UUIDs** in a recognizable namespace (e.g. fixtures
`5eed0000-…`) so reset targets them precisely without touching the baseline.

## 3. Fixture states — all four at once, no cron (§"how")
One **`open`** round `R_open` (window contains now). Grouping is **per-fixture** via
`canPredict`, so a single open round yields all four groups purely from each
fixture's kickoff/status — no lifecycle cron required:
| Fixture | Teams | kickoff | status | QA prediction | Group |
|---|---|---|---|---|---|
| F_stp | Arsenal v Liverpool | now **+3d** | scheduled | none | **Still To Predict** |
| F_pred | Man City v Chelsea | now **+2d** | scheduled | 1–1 | **Predicted** |
| F_locked | Spurs v Man Utd | now **−6h** | scheduled (kicked off) | 2–0 | **Locked** |
| F_done | Real Madrid v Barcelona | now **−1d** | **finished 2–1** | 2–1 | **Completed** |

`R_next` (status `draft`, starts next Thursday) holds 2 future scheduled fixtures
(Bayern v Dortmund, PSG v Juventus) — used to verify the **predict gate rejects**
predictions on a non-open round (§5/§6 of the checklist).

## 4. Predictions → scoring (§10) on F_done (actual 2–1)
| User | F_done pick | Result | F_pred | F_locked |
|---|---|---|---|---|
| QA | 2–1 | **exact = 5** | 1–1 (unscored) | 2–0 (unscored, locked) |
| comp1 | 1–0 | **correct outcome = 3** | 0–0 | 1–1 |
| comp2 | 0–2 | **wrong = 0** | 2–2 | 0–1 |
F_stp deliberately **unpredicted** by QA (covers the unpredicted case).
After `recalculate_match_predictions(F_done)`: points are 5/3/0. Resulting standings
(one scored fixture): **QA #1 (5), comp1 #2 (3), comp2 #3 (0)** — identical for
Round, Season, and All-Time, and within each league.

## 5. leaderboard_entries (§11) via `recalculate_leaderboards(R_open)`
Generates: global **Round** `(R_open, season, null)`, global **Season**
`(null, season, null)`, and **League Round/Season** for the two non-global leagues.
Global-League board reads the `league_id IS NULL` rows. **All-Time** is computed at
query time over the season rows — appears automatically. (Idempotent: the
`0014` unique index means re-running yields identical rows.)

## 6. Leagues (§12)
- **L_private** "QA Private League" (`visibility=private`, invite_code fixed) — members QA, comp1, comp2.
- **L_public** "QA Public League" (`visibility=public`) — members QA, comp1 (leave comp2 out to test join).
- Global League — all four via auto-join (admin excluded from leaderboards by RPCs).

## 7. Admin / sync test data (§14, §17, §18)
- `system_alerts`: one **unresolved** (`is_read=false, resolved_at=null`, e.g.
  `sync_stale`) + one **resolved** (`is_read=true, resolved_at set`). → unread badge
  = 1; Resolve button testable.
- `sync_logs`: last successful `fixture_discovery` **> 12h ago** (lights the stale
  banner) + a couple of `failed` rows (recent-runs table).
- Context-create: the active standard 2026-27 context exists → creating another
  **active** standard context is **rejected**; creating an **upcoming** one against
  **2027-28** succeeds (hence seeding the 2nd season).
- Maintenance mode (§15) is **out of the seed** — toggled via Edge Config at QA time.

## 8. /play state matrix (§1–§3) — toggles, not separate data
Default seed → **active** state (R_open open). To exercise the other two without
re-seeding, run the documented one-line UPDATEs on staging, reload `/play`, revert:
- **coming_up:** set `R_open.status='finalized'` (or `empty`) → only the draft
  `R_next` remains → `/play` shows coming_up.
- **summer gap:** also set `R_next.status='cancelled'` (or `empty`) → no
  open/in_progress and no upcoming → `/play` shows "No active LeagueXI Round".
- Revert: `R_open.status='open'`, `R_next.status='draft'`.
(Status-driven, per §11 of the spec — no calendar edits.)

## 9. Staging-only safety
The script refuses to run unless **all** hold:
- `SEED_ENV=staging` explicitly set, **and**
- a `--yes-staging` CLI flag, **and**
- `SUPABASE_URL` matches a configured **staging project ref** (and is **not** the
  known production ref — hard denylist).
It uses only the **staging** service-role key from env. No production credentials are
ever referenced or committed. The script lives under `scripts/` and is never imported
by the app/build.

## 10. Reset / safe re-run
- **Idempotent re-run:** all app rows use fixed UUIDs → upsert (`on conflict do
  update`); users looked up by email and reused (Admin API create is not idempotent).
  Re-running refreshes time-relative kickoffs (see Risks) and re-derives points/boards.
- **`--reset` mode:** deletes only seeded rows, FK-safe order: `leaderboard_entries`
  (seeded round/leagues) → `predictions` (seeded fixtures) → `league_members` →
  `leagues` (seeded) → `fixtures` (seeded) → `system_alerts`/`sync_logs` (seeded ids)
  → `leaguexi_rounds` (R_open, R_next) → `competitions` (seeded) → `seasons` (2027-28)
  → seeded `auth.users` (Admin API `deleteUser`, cascades profiles/predictions/
  memberships). **Never deletes** the baseline (active context, 2026-27 season, 15
  clubs, Global League).

## 11. Assumptions about auth/profiles
- Supabase Auth (GoTrue) owns `auth.users`; the seed uses the Admin API + service
  role; `on_auth_user_created` creates `profiles`; setting `username` triggers Global
  League auto-join — so the **Global League must exist** on staging (baseline).
- Staging email confirmation is bypassed (`email_confirm:true`).
- `push-notifications.sql` is irrelevant to UI QA (push is native/dormant).

## 12. Risks
- **Time-relative kickoffs:** F_stp/F_pred (future) and F_locked (past) depend on
  `now()` at seed time. If QA runs much later, states drift → **re-run the seed
  same-day** before a QA session. (No way to freeze the server clock.)
- **Wrong-DB risk:** mitigated by the §9 triple guard; operator must point env at
  staging only.
- **Team crests are initials** (`logo_url` null) — explicit, not a bug (§16).
- `competitions` requires NOT-NULL `slug/season/starts_at/ends_at` → the seed supplies
  them for the staging competition.
- An `open` round containing a `finished` fixture is slightly unnatural (cron would
  move it to `in_progress`) but valid for QA since grouping is per-fixture.
- Deleting seeded auth users cascades — ensure no real/baseline user shares the
  seeded emails (they use a `.test` domain to avoid collisions).

## 13. Verification steps (after seeding)
- Row counts: 4 users, 2 seasons, 1 competition, 2 rounds, 6 fixtures, ≥9 predictions,
  2 leagues (+global memberships), leaderboard_entries > 0, 2 alerts, ≥3 sync_logs.
- `/play` (as QA): active card, progress (e.g. 3/4), Still-To-Predict shows F_stp.
- `/rounds/current` → R_open; groups show F_stp / F_pred / F_locked / F_done correctly.
- `/leaderboards`: Season default QA#1 comp1#2 comp2#3; Round + All-Time match; round selector lists R_open.
- `/leagues/[L_private]` + `[L_public]`: Round/Season/All-Time/Members tabs; QA rank in header.
- `/profile` (QA): Total 5, Exact 1, Correct 0, Accuracy 100%, Season #1, All-Time #1, 3 leagues.
- `/admin/sync` (admin): unread badge =1, stale banner on, Resolve works; `/admin/contexts`: active-create rejected, upcoming-create (2027-28) succeeds.
- Run the §8 toggles to confirm coming_up + gap.
